7) CADEIA OFICIAL - QUEM CRIA QUEM
====================================================================

Regra fonte: ROLE_MANAGEMENT_ALLOWED (apps/core/rbac.py)

7.1 ADMIN
- Pode criar todos os 43 perfis.

7.2 MUNICIPAL
- Pode criar 41 perfis.
- Nao pode criar ADMIN nem MUNICIPAL.

7.3 SECRETARIA
- Pode criar 25 perfis operacionais/de unidade.
- Inclui criacao de UNIDADE e perfis finais de operacao.

7.4 UNIDADE
- Pode criar 18 perfis operacionais finais.

7.5 Observacao de governanca importante
- Perfis com permissao accounts.manage_users: ADMIN, MUNICIPAL, SECRETARIA, UNIDADE, NEE, CAD_GESTOR.
- Porem, cadeia de atribuicao formal so existe para: ADMIN, MUNICIPAL, SECRETARIA, UNIDADE.
- Isso significa que NEE e CAD_GESTOR podem acessar gestao de usuarios, mas nao atribuem papel na cadeia padrao.
- Recomendacao: manter assim somente se for intencional (operacao de cadastro sem delegacao de perfis).


====================================================================
8) ESCOPOS POR NIVEL (FUNIL PROFISSIONAL)
====================================================================

8.1 Nivel 0 - ADMIN global
- Escopo: total
- Cria: tudo

8.2 Nivel 1 - MUNICIPAL
- Escopo minimo: municipio
- Cria: gestores de secretaria, perfis especializados municipais, operacionais amplos

8.3 Nivel 2 - SECRETARIA
- Escopo minimo: municipio + secretaria
- Cria: unidades e perfis de unidade/operacao

8.4 Nivel 3 - UNIDADE
- Escopo minimo: municipio + secretaria + unidade
- Cria: perfis finais de execucao

8.5 Nivel 4 - PERFIS OPERACIONAIS
- Escopo: unidade/setor/local, conforme caso
- Nao devem criar perfis de governanca

8.6 Nivel 5 - USUARIOS DE IDENTIDADE (ALUNO/PACIENTE/CIDADAO)
- Conta individual de portal
- Vinculo obrigatorio ao sujeito de negocio:
  - ALUNO -> aluno
  - CIDADAO (paciente) -> paciente (e opcionalmente aluno vinculado)


====================================================================
9) PROCESSO PROFISSIONAL DE CRIACAO DE USUARIO
====================================================================

9.1 Usuario administrativo (funcional)
1. Definir papel alvo (role) conforme matriz.
2. Definir escopo organizacional exigido.
3. Preencher dados basicos (nome, email, CPF).
4. Salvar conta.
5. Sistema:
   - gera codigo_acesso unico;
   - define senha inicial como CPF (somente digitos);
   - ativa must_change_password;
   - grava auditoria CREATE.

9.2 Usuario tipo ALUNO
1. Selecionar role = ALUNO.
2. Selecionar aluno vinculado (obrigatorio).
3. Informar escopo coerente.
4. Salvar.
5. Sistema passa a permitir escopos de portal do proprio aluno.

9.3 Usuario tipo PACIENTE (via CIDADAO)
1. Selecionar role = CIDADAO.
2. Selecionar paciente vinculado (obrigatorio no fluxo recomendado).
3. Se paciente tiver aluno associado, sistema pode aproveitar vinculo educacional.
4. Salvar com escopo coerente.

9.4 Edicao de usuario
- Mantem mesmas regras de cadeia, escopo e validacao.
- Gera auditoria UPDATE.

9.5 Bloqueio/inativacao/reset
- Toggle ativo/inativo: auditoria ACTIVATE/DEACTIVATE
- Toggle bloqueio: auditoria BLOCK/UNBLOCK
- Reset codigo: RESET_CODE
- Reset senha para CPF: RESET_PASSWORD + must_change_password=True


====================================================================
10) SEGURANCA E CONFORMIDADE
====================================================================

10.1 Senha e primeiro acesso
- Senha inicial padrao por CPF (digitos).
- Troca obrigatoria no primeiro login.

10.2 MFA
- mfa_enabled disponivel no profile.
- Fluxo de codigo temporario com limite de tentativas.

10.3 Politica de expiração
- password_expires_days configuravel.
- perfil expira e exige troca quando necessario.

10.4 Protecao de dados pessoais
- CPF mascarado em campo visivel.
- CPF hash e criptografia para seguranca.

10.5 Auditoria
- Toda acao critica de usuario e registrada.
- Base para rastreabilidade e compliance.

====================================================================
12) GUIA PRATICO PARA IMPLANTACAO (PREFEITURA)
====================================================================

12.1 Sequencia recomendada de ativacao
1. Criar gestor MUNICIPAL.
2. Cadastrar secretarias base:
   - Gestao/Administracao
   - Educacao
   - Saude
3. Criar gestor SECRETARIA por secretaria.
4. Criar unidades (escolas, UBS, etc.).
5. Criar gestor UNIDADE.
6. Criar perfis operacionais por unidade.
7. Criar contas de aluno/paciente quando houver portal ativo.

12.2 Quem deve criar no dia a dia
- Novas secretarias: MUNICIPAL
- Novas unidades: SECRETARIA
- Novos operadores locais: UNIDADE
- Contas de aluno/paciente: SECRETARIA ou UNIDADE (conforme governanca local)

12.3 Matriz rapida (uso recomendado)
- MUNICIPAL: governanca, planejamento, controle e criacao estrutural
- SECRETARIA: gestao do modulo da pasta e expansao de unidades
- UNIDADE: operacao diaria e equipe de ponta
- ALUNO/CIDADAO: acesso individual ao proprio contexto

13) PERFIS QUE MERECEM DECISAO DE PRODUTO (PENDENCIAS)
====================================================================

13.1 Papeis de Camara no motor RBAC
- Existem no motor, mas nao estao em Profile.Role (nao cadastraveis pela tela).
- Decidir:
  - incluir em Profile.Role para cadastro direto, ou
  - manter apenas para uso interno/legado.

13.2 manage_users x cadeia de atribuicao
- NEE e CAD_GESTOR possuem accounts.manage_users, mas nao possuem chave na cadeia formal.
- Decidir se:
  - entram oficialmente como criadores com regras especificas, ou
  - permanecem apenas para operacao auxiliar sem delegacao de perfis.

13.3 Portal paciente dedicado
- Com vinculo profile.paciente, a identidade de paciente esta estruturada.
- Proximo passo de produto (se desejado):
  - desenhar permissoes finas para portal de paciente sem expor telas administrativas.

18) CONCLUSAO
====================================================================

O sistema de usuarios do GEPUB esta pronto para operar em cadeia profissional
com governanca por nivel, escopos definidos e trilha de auditoria.

Com as melhorias desta revisao, o processo de criacao de usuario passa a cobrir
formalmente:
- usuario funcional (administrativo/operacional),
- usuario aluno,
- usuario paciente (via perfil cidadao com vinculo de paciente).

Este arquivo pode ser usado como referencia oficial para reorganizacao final
(de nomenclatura, fusao de perfis e desenho definitivo da tela de usuarios)
sem perder rastreabilidade tecnica.
